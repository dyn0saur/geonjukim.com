# Research

## Project concept

- 개인 포트폴리오 전체를 Grasshopper 스타일의 하나의 큰 노드 캔버스로 구성한다.
- 실제 파라메트릭 계산 엔진은 만들지 않는다. 각 프로젝트는 미리 정의된 입력, 연결 액션, 상태 변화, 결과를 갖는 고정 시나리오로 재생한다.
- 장기적으로 초기 조건 3D → 해체/분석을 상징하는 컴포넌트 → 작업 및 조건 입력 → 결과 3D의 흐름으로 프로젝트를 설명한다.
- 결과 노드에는 향후 Three.js 기반 Rhino 모델 뷰어를 포함할 수 있어야 한다.

## Phase 1 problem definition

Phase 1은 전체 포트폴리오 제작이 아니라 핵심 인터랙션의 기술적 성립 여부를 검증하는 프로토타입이다.

검증 시나리오:

1. 숫자 패널 `5`, `3`, 덧셈을 상징하는 컴포넌트, 결과 패널을 배치한다.
2. 초기에는 `3 → B`만 연결하고 `A`는 비워 둔다.
3. A 입력이 없을 때 컴포넌트는 경고 색상과 경고 표시를 사용한다.
4. 경고 표시에 마우스를 올리면 우측 상단에 `A가 입력되지 않았습니다` 메시지를 표시한다.
5. 사용자가 `5` 패널의 출력 포트에서 A 입력 포트까지 드래그하면 임시 Bézier 와이어가 포인터를 따른다.
6. 유효 포트에 드롭하면 와이어가 연결되고 컴포넌트가 정상 상태로 바뀌며 결과 패널의 표시가 `3`에서 `8`로 바뀐다.
7. 이 변화는 실제 덧셈이 아니라 `missing-input → connected → completed`로 정의된 고정 상태 전이다.

## Feasibility assessment

- 구현 가능성이 높다. 노드 위치와 연결 관계를 데이터로 관리하고, 뷰 변환과 상태 전이를 분리하면 계산 엔진 없이도 요구된 동작을 구현할 수 있다.
- 노드 본체는 HTML 요소, 와이어와 임시 연결선은 같은 월드 좌표계를 공유하는 SVG 레이어로 구성하는 방식이 Phase 1에 적합하다.
- 이동, 확대/축소, 스냅은 모든 노드의 좌표를 직접 다시 쓰기보다 하나의 월드 변환으로 처리하는 편이 일관적이다.
- 와이어는 시작/끝 포트의 월드 좌표로부터 cubic Bézier 경로를 계산할 수 있다. 노드 이동이나 확대/축소 시 같은 좌표 변환을 적용한다.
- 향후 Three.js 뷰어는 노드 내부의 독립 렌더링 영역으로 추가할 수 있다. 노드 캔버스의 카메라와 3D 카메라의 휠/드래그 이벤트 충돌 정책은 별도로 필요하다.

## Proposed state model

- `Node`: 고유 ID, 종류, 월드 좌표, 시각 상태, 입력/출력 포트
- `Connection`: 시작 포트 ID, 끝 포트 ID, 연결 상태
- `Scenario`: 허용된 액션과 그 액션이 유발하는 노드/결과 상태 변경
- `Viewport`: pan x/y, zoom
- `Interaction`: idle, panning, moving-node, connecting

계산값을 유도하지 않고 시나리오의 명시적 전이로 결과를 바꾼다. 예: `connect(panel-5.out, add.A)`가 성공하면 `add.status = normal`, `result.text = 8`.

## Risks and decisions still needed

- 경고 말풍선이 hover 전용이면 터치와 키보드 사용자가 접근할 수 없으므로 focus/click 대체 동작을 고려해야 한다.
- Three.js 노드 위에서 포인터를 조작할 때 3D orbit과 캔버스 pan/zoom 중 어느 쪽이 우선하는지 정의해야 한다.

## Confirmed Phase 1 decisions

- 격자 한 칸은 월드 좌표 기준 가로 15단위, 세로 5단위다.
- 스냅은 격자 스냅이 아니다. 이동 중 인접 컴포넌트의 상·하·좌·우 외곽, 가로·세로 중심, 입력·출력 포트의 수평축에 정렬한다.
- pan은 마우스 우클릭 드래그로 조작한다.
- 확대/축소 범위는 20%에서 300%다.
- 300% 최대 확대에서 월드 1단위가 1 CSS px에 대응한다.
- Phase 1 구현은 HTML 노드와 SVG Bézier 와이어를 하나의 월드 변환 아래 결합하는 구조로 성립했다. 브라우저별 시각 및 포인터 QA는 후속 검증 대상으로 남긴다.
- 배경 그리드의 시각 셀 간격은 기존 15×5단위의 40배인 600×200 월드 단위로 확대한다.
- 연결 드래그는 포인터 방향의 삼각 화살표를 가진 임시 점선 와이어로 표시한다. 연결된 `5 → A` 와이어는 Ctrl을 누른 채 같은 경로로 드래그하여 삭제할 수 있다.

## Hosting strategy

- 프로토타이핑과 초기 포트폴리오 개발은 ChatGPT Sites에서 계속한다.
- ChatGPT Sites는 개발·검토·초기 공개를 위한 현재 호스팅 환경이지, 애플리케이션 핵심 구조가 의존해야 하는 영구 플랫폼으로 간주하지 않는다.
- 콘텐츠와 기능이 안정된 뒤 Cloudflare 등 독립 호스팅 후보를 평가하고, 선택한 환경으로 이전하는 별도 단계를 수행한다.
- 이전 시점까지 캔버스, 시나리오 상태, Three.js 렌더링, 프로젝트 콘텐츠 및 정적 자산은 호스팅 제공자와 무관한 표준 웹 코드와 데이터로 유지한다.
- Sites 전용 배포 메타데이터, 저장소 연결, 환경 변수 및 런타임 바인딩은 핵심 애플리케이션과 분리한다.
- 최종 호스팅 후보는 실제 트래픽, 3D 자산 용량, 캐시 요구, 비용, 사용자 지정 도메인, 분석 및 배포 운영 요구가 드러난 뒤 결정한다.
- 프로토타이핑 기간에는 검증을 통과한 구현 수정마다 현재 비공개 ChatGPT Sites 배포를 갱신한다. 사용자가 로컬 전용을 요청하거나 게시 권한이 없는 경우에만 예외로 한다.

## Phase 2-1 — HANA HQ project canvas

- HANA HQ를 첫 실제 포트폴리오 시나리오로 좌에서 우로 배치한다. 실제 Three.js 또는 2D 도면 렌더링은 아직 포함하지 않으므로 Phase 3이 아니라 Phase 2의 첫 세부 단계로 분류한다.
- 흐름은 제목 `HANA HQ` → `Ref. SRF` → `DECONSTRUCT REF. SURFACE` → `FABRICATION & CONSTRUCTION REQUIREMENTS` 및 `CONSTRUCT WOOD PANEL 3D` → `GENERATE 2D DRAWINGS` 순서다.
- 캔버스 요소는 제목 Scribble, 모델/이미지 뷰어, 연결 컴포넌트, 텍스트·숫자 패널, 입출력 포트를 가진 주요 컴포넌트로 구분한다.
- Deconstruct의 출력은 `Wood Panel`, `Wood Block`, `Corner Reveal`, `Base Board`, `T-Bar System`이며 Phase 2-1의 후속 상호작용은 `Wood Panel`만 대상으로 한다.
- Fabrication 출력은 `Wood veneer application`, `Laminated timber fabrication followed by steam bending`, `20`, `200`, `6`, `2400` 패널을 거쳐 Construct 입력으로 전달한다.
- Construct의 `3D Model`과 Generate의 `2D Drawing`은 후속 Three.js 및 이미지 연결을 위한 빈 뷰어 자리만 제공한다.
- 주요 컴포넌트 이름은 Bifocals 플러그인처럼 컴포넌트 위 말풍선형 라벨로 항상 표시한다.
- 컴포넌트 아이콘은 제공된 Grasshopper 스크린샷을 스타일 참조로 삼아 독자적인 PNG 자산으로 생성하고 프로젝트 안에 보관한다.

### Confirmed Phase 2-1 decisions

- Ref. SRF→Deconstruct, Wood Panel→Construct, Construct→3D viewer, 3D Model 연결 컴포넌트→Generate의 네 연결은 사용자가 완성하며 나머지 지정 연결은 초기 상태부터 존재한다.
- Generate 출력은 2D 전용 이미지 뷰어에 연결한다.
- 화면 표기는 `REQUIREMENTS`, `WOOD PANEL`, `Ref. SRF`처럼 교정된 영문 철자를 사용한다.
- 생성 PNG 아이콘은 `DECONSTRUCT REF. SURFACE`, `FABRICATION & CONSTRUCTION REQUIREMENTS`, `CONSTRUCT WOOD PANEL 3D`, `GENERATE 2D DRAWINGS`의 네 주요 컴포넌트에만 사용한다. 연결 컴포넌트는 아이콘 없이 이름만 표시한다.
- 최초 viewport는 제목과 초기 단계를 읽을 수 있게 시작하고 사용자가 우측으로 pan하여 전체 흐름을 탐색한다.
- 제목 Scribble도 다른 노드처럼 클릭 선택과 드래그 이동이 가능하며, 선택 시 짙은 초록색 글자로 표시한다.
- Phase 2-1부터 노드와 연결을 JSX에 개별 하드코딩하지 않고 프로젝트 시나리오 데이터와 범용 렌더러를 분리한다.

### Phase 2-1 implementation result

- `model.ts`의 검증된 공통 스키마, `hana-hq.ts`의 프로젝트 데이터, `node-canvas.tsx`의 범용 렌더러로 계층을 분리했다. 중복 노드·포트·연결, 잘못된 포트 방향, 하나의 입력에 대한 중복 연결은 시나리오 정의 시 오류로 처리한다.
- 초기 고정 연결 20개와 사용자 완성 연결 4개를 데이터로 선언했다. 사용자는 Ref. SRF, Wood Panel, 3D viewer, Generate 2D 입력을 순서와 무관하게 연결하거나 Ctrl+드래그로 삭제할 수 있다.
- 필수 입력 상태로 주요 컴포넌트의 경고를 도출하고, 3D/2D 뷰어는 관련 연결 완료 시 준비 placeholder로 전환한다.
- 네 주요 컴포넌트에 독자적인 투명 PNG 아이콘을 적용했고 연결 컴포넌트는 이름만 표시한다.
- 1920×1080 브라우저 QA에서 초기 좌→우 레이아웃, Scribble 선택·이동, 선택 와이어 그라데이션, 0/4→4/4 연결, 경고 해제와 뷰어 준비 전환을 확인했다.
- 로컬 `vinext start`는 이 환경에서 HTML을 반환했지만 `/assets/*`를 404로 반환했다. Vite 개발 서버에서는 정적 자산과 전체 인터랙션이 정상이며 Sites 배포 후 자산 응답을 다시 확인해야 한다.
- 참고 디자인 재검토 후 연결 컴포넌트는 텍스트에 밀착한 얇은 블록, 제작값 패널은 무그라데이션 단색과 1 CSS px 반투명 흰색 inner stroke, 주요 컴포넌트는 입력·아이콘·출력의 공통 3열 구조로 정리했다.
- 모델·이미지 뷰어의 기본 가로·세로를 기존 데이터의 2.5배로 확장하고, 네 모서리 자유 리사이즈 상태를 위치·크기 데이터와 분리해 와이어 좌표가 실시간 추종하도록 구현했다.
- 빈 캔버스 좌클릭 드래그를 Fence 선택으로 정의했다. 교차한 노드는 선택 집합이 되며, 집합의 한 노드를 드래그하면 스냅된 동일 이동량을 전체에 적용해 상대 위치를 유지한다.
- 포트는 노드 경계 뒤에 원의 절반이 가려진 반원으로 통일했고, 사용자 연결과 Ctrl 연결 해제의 실제 포인터 적중은 전체 원 크기의 투명 버튼으로 유지한다.
- 네 주요 처리 아이콘을 imagegen으로 다시 제작한 뒤 투명 배경과 제한 단색 팔레트로 정규화해 단순한 도형·색·검정 외곽선만 남겼다.
- 주요 컴포넌트의 크기와 포트 행은 입력·출력 라벨 길이, 중앙 아이콘 폭, 열 간격, 상하 패딩으로 계산한다. 이에 따라 입력은 좌측 정렬되고 콘텐츠가 바뀌어도 본체 크기가 데이터에서 함께 갱신된다.
- 하나의 `Wood Panel` 연결 컴포넌트 출력을 Construct용·Fabrication용 두 연결 컴포넌트로 명시적으로 분기하고, 두 경로의 상태와 와이어를 독립 연결 데이터로 관리한다.
- 와이어는 X 0.5, Y 0.75 비율의 cubic Bézier 제어점을 사용한다. 텍스트와 아이콘은 100%에서 70%까지 선형으로 페이드되고 70% 미만에서 숨겨지는 확대 단계별 표현을 적용했다.
- 1920×1080 Chrome QA에서 포트 뒤쪽 레이어, 콘텐츠 기반 레이아웃, 두 Wood Panel 분기, 보이지 않는 코너 리사이즈, Bézier 제어점과 확대 단계별 불투명도를 검증했다.
- 편집 가능한 위치·크기·활성 연결을 하나의 캔버스 문서 스냅샷으로 묶고 최대 100단계의 undo/redo 히스토리로 관리한다. 노드·그룹 이동과 뷰어 리사이즈는 포인터 이동 횟수와 무관하게 드래그 종료 시 한 액션으로 기록하며, 연결 추가·삭제도 동일한 히스토리에서 복원한다.
- Chrome QA에서 `Ctrl+Z`/`Ctrl+Y`로 이동·리사이즈·연결 추가·연결 삭제를 왕복하고, undo 후 새 편집 시 기존 redo 스택이 폐기되는 것을 검증했다.
- Alt+좌클릭 수평 드래그를 기준선 기반 캔버스 분리 액션으로 추가했다. 우측 드래그는 기준선 오른쪽에 완전히 놓인 노드만 우측으로, 좌측 드래그는 왼쪽 노드만 좌측으로 이동하며 기준선을 가로지르는 노드는 유지한다.
- 분리 드래그 중 원래 기준선과 현재 경계 사이를 반투명 밴드·점선·방향 화살표로 표시한다. Chrome QA에서 양방향 확장, 한 드래그 중 방향 반전, 교차 와이어 추종과 Ctrl+Z/Y 복원을 검증했다.
